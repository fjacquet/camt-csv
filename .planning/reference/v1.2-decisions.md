# v1.2 Decisions Log

All decisions made during v1.2 milestone planning (2026-02-15).

## Scope Decisions

### D-01: Full Polish scope (not just Revolut)
**Decision:** v1.2 covers Revolut overhaul + cross-parser consistency + AI safety + batch everywhere
**Context:** User chose "Full polish" over narrower Revolut-only or Revolut+consistency options
**Impact:** 18 requirements across 5 categories, 4 phases

### D-02: Include all Revolut transaction types in output
**Decision:** No filtering — all 8 types (Transfer, Card Payment, Exchange, Deposit, Fee, Charge, Card Refund, Charge Refund) appear in output
**Context:** User explicitly chose "Include all" when asked about transfers and exchanges
**Impact:** Parser must tag each type; iCompta import gets everything; user filters in iCompta

### D-03: Include Exchange transactions as-is
**Decision:** Exchange transactions are kept visible with type tag, not hidden or filtered
**Context:** User chose "Include as-is" — currency conversions are real financial events
**Impact:** No need for exchange-pair matching in v1.2; each file stands alone

### D-04: No cross-file exchange pairing in v1.2
**Decision:** Each Revolut CSV file is processed independently; no attempt to match EUR↔CHF exchange pairs across files
**Context:** Pairing requires ±timing tolerance, ±rounding tolerance, multi-currency matching — too complex for this milestone
**Impact:** Exchange appears in each file independently; cross-file reconciliation deferred to future

## Architecture Decisions

### D-05: Target iCompta as import destination
**Decision:** CSV output is designed to import into iCompta macOS personal finance app
**Context:** User provided iCompta SQLite database; schema analyzed; import plugins already configured
**Reference:** `.planning/reference/icompta-schema.sql`, `.planning/reference/icompta-import-plugins.txt`
**Impact:** CSV column names, separator, date format must match existing iCompta import plugin configuration

### D-06: Output formatter is a cross-parser feature
**Decision:** `--format` flag (standard, icompta) works on ALL parsers (CAMT, PDF, Revolut, Selma, Debit), not just Revolut
**Context:** User asked "so the other parsers can not be mapped to new process?" — confirmed all parsers should benefit
**Impact:** OutputFormatter sits between Transaction model and CSV writer; any parser's output can use any format

### D-07: Match existing iCompta import plugins
**Decision:** Use the exact column names, separator, and date format from the user's existing iCompta import plugins
**Context:** Queried user's iCompta SQLite database; found CSV-Revolut-CHF, CSV-Revolut-EUR, CSV-RevInvest plugins already configured
**Specifics:**
- **Separator:** Semicolon (`;`)
- **Date format:** `dd.MM.yyyy`
- **Encoding:** UTF-8
- **Header:** Yes
- **Key columns:** Date, ValueDate, Description, Amount, CreditDebit, Type, Status, Category, BookkeepingNumber, Fees, Fund, NumberOfShares, InvestmentType
**Impact:** No need to create new iCompta import plugins; our CSV output must match these column names exactly

### D-08: Other parsers are NOT broken by Revolut changes
**Decision:** CAMT, PDF, Selma, Debit parsers are untouched by Revolut-specific work
**Context:** These parsers already output 35-column standardized CSV; only Revolut is behind (4-column)
**Impact:** Risk is isolated to Revolut parser internals + new output formatter option

## Data Model Decisions

### D-09: Separate iCompta account for Revolut Savings
**Decision:** Create "Revolut CHF Vacances" as a separate Savings account in iCompta
**Context:** CHF file contains 688 Savings transactions (Product=Savings) — all "To pocket CHF Vacances" transfers; user confirmed this should be a separate account
**Impact:** Parser must route transactions by Product+Currency combination

### D-10: Account routing by Product+Currency
**Decision:** Revolut transactions route to iCompta accounts as follows:
- `Product=Current` + `Currency=CHF` → **Revolut CHF**
- `Product=Savings` + `Currency=CHF` → **Revolut CHF Vacances**
- `Product=Current` + `Currency=EUR` → **Revolut EUR**
**Context:** Derived from iCompta account list + Revolut CSV Product field analysis
**Impact:** Output CSV may need an "Account" column or separate output files per account

### D-11: AI auto-learn defaults to OFF
**Decision:** `--auto-learn` flag controls whether AI categorizations are saved to YAML; defaults to disabled
**Context:** Research identified risk of AI miscategorization becoming permanent immediately; user must opt-in
**Impact:** Existing behavior changes — current auto-save becomes opt-in; safer but requires flag for learning
